Modelet vs MyBatis & Hibernate

Hibernate hides all your SQL. MyBatis makes you write all of it. Modelet splits the work where it belongs.

All three answer the same question — how does an object meet a row? They just disagree about who writes the SQL. Here is the honest comparison.

Three philosophies

Who writes your SQL?

Hibernate

Hides all SQL

You describe mappings and relations; the framework derives SQL at flush time. Powerful for complex domain graphs — at the price of HQL/JPQL, session lifecycles, lazy proxies and dirty-checking surprises.

MyBatis

Makes you write all SQL

Full control: every SELECT — and every INSERT, UPDATE, DELETE — is written by hand in mapper XML or annotations, plus a resultMap per shape. Honest, but the write side is pure toil.

Modelet

You write only the queries

SELECTs are plain ANSI SQL in your code. The write side is generated from the entity's TxnMode state — key written back, audit fields stamped. ~2,000 lines; read it in an afternoon.

The decisive difference

The write side is where lives diverge

Same requirement in all three: insert a book, then raise its price. Watch who does the typing.

Modelet2 lines per write
Book book = new Book();
book.setTitle("Modelet in Action");
model.save(book);                  // INSERT — id written back, audit stamped

book.setPrice(new BigDecimal("499"));
model.save(book);                  // UPDATE — same call, TxnMode decides
MyBatisevery column, by hand
<!-- BookMapper.xml — you write and maintain every statement -->
<insert id="insertBook" useGeneratedKeys="true" keyProperty="id">
  INSERT INTO book (bookName, price) VALUES (#{title}, #{price})
</insert>
<update id="updateBook">
  UPDATE book SET bookName = #{title}, price = #{price} WHERE id = #{id}
</update>

// Java — plus a mapper interface to keep in sync
bookMapper.insertBook(book);
bookMapper.updateBook(book);
Hibernatestate machine you must learn
session.persist(book);             // transient → managed
book.setPrice(new BigDecimal("499"));
// no save call — dirty checking emits an UPDATE at flush… eventually.
// detached instance? use merge(). outside the session? LazyInitializationException.

A new column in MyBatis touches the table, the XML insert, the XML update and the mapper. In Modelet it touches the table and the entity field — the SQL regenerates itself.

Honest notes

When the others win

Hibernate

A deep domain model with complex object graphs, vendor portability and second-level caching — and a team already fluent in JPA. That is Hibernate's home turf.

MyBatis

You need hand-tuned SQL on the write side too — bulk upserts, vendor hints, stored procedures — or its dynamic SQL tags (<if>, <foreach>) fit your query patterns.

Modelet

Your team is fluent in SQL and wants to ship on day one: queries stay in SQL you control, writes generate themselves, and the whole framework fits in your head.
