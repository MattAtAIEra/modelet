Modelet vs MyBatis 與 Hibernate

Hibernate 把 SQL 全藏起來,MyBatis 要你把 SQL 全寫完。Modelet 把工作切在該切的地方。

三者回答的是同一個問題 —— 物件如何對上資料列?分歧只在:SQL 該由誰來寫。以下是誠實的比較。

三種哲學

你的 SQL,誰來寫?

Hibernate

把 SQL 全部藏起來

你描述 mapping 與關聯,框架在 flush 時推導出 SQL。對複雜的領域物件圖很強大 —— 代價是 HQL/JPQL、Session 生命週期、lazy proxy 與 dirty checking 的意外。

MyBatis

要你把 SQL 全部寫完

完全掌控:每一句 SELECT —— 連同每一句 INSERT、UPDATE、DELETE —— 都要在 mapper XML 或註解裡手寫,外加每種結果形狀一份 resultMap。誠實,但寫入面純屬苦工。

Modelet

你只寫查詢

SELECT 就是程式碼裡的 ANSI SQL。寫入面由 entity 的 TxnMode 狀態自動產生 —— 主鍵回填、稽核欄位自動蓋上。約 2,000 行,一個下午讀得完。

寫入面,是人生分歧的地方

同一個需求:新增一本書,然後調價。看看是誰在打字。

Modelet2 lines per write
Book book = new Book();
book.setTitle("Modelet in Action");
model.save(book);                  // INSERT — id written back, audit stamped

book.setPrice(new BigDecimal("499"));
model.save(book);                  // UPDATE — same call, TxnMode decides
MyBatisevery column, by hand
<!-- BookMapper.xml — you write and maintain every statement -->
<insert id="insertBook" useGeneratedKeys="true" keyProperty="id">
  INSERT INTO book (bookName, price) VALUES (#{title}, #{price})
</insert>
<update id="updateBook">
  UPDATE book SET bookName = #{title}, price = #{price} WHERE id = #{id}
</update>

// Java — plus a mapper interface to keep in sync
bookMapper.insertBook(book);
bookMapper.updateBook(book);
Hibernatestate machine you must learn
session.persist(book);             // transient → managed
book.setPrice(new BigDecimal("499"));
// no save call — dirty checking emits an UPDATE at flush… eventually.
// detached instance? use merge(). outside the session? LazyInitializationException.

在 MyBatis 加一個欄位,要動資料表、XML 的 insert、XML 的 update、還有 mapper。在 Modelet 只要動資料表和 entity 欄位 —— SQL 自己重新長出來。

誠實備註

什麼時候該選別人

Hibernate

深層領域模型、複雜物件圖、跨資料庫可攜性與二級快取 —— 而且團隊已經熟 JPA。那是 Hibernate 的主場。

MyBatis

連寫入面都需要手工調校的 SQL —— 批次 upsert、廠商 hint、預存程序 —— 或者它的動態 SQL 標籤(<if>、<foreach>)正好貼合你的查詢模式。

Modelet

團隊熟 SQL、想第一天就交付:查詢留在你掌控的 SQL 裡,寫入自動產生,整個框架裝得進腦袋。
