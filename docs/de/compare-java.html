Modelet vs. MyBatis & Hibernate

Hibernate versteckt dein ganzes SQL. MyBatis lässt dich alles selbst schreiben. Modelet teilt die Arbeit dort, wo sie hingehört.

Alle drei beantworten dieselbe Frage — wie trifft ein Objekt auf eine Zeile? Sie sind sich nur uneins, wer das SQL schreibt. Hier der ehrliche Vergleich.

Drei Philosophien

Wer schreibt dein SQL?

Hibernate

Versteckt alles SQL

Du beschreibst Mappings und Relationen; das Framework leitet SQL beim Flush ab. Stark für komplexe Domänengraphen — zum Preis von HQL/JPQL, Session-Lebenszyklen, Lazy-Proxies und Dirty-Checking-Überraschungen.

MyBatis

Lässt dich alles SQL schreiben

Volle Kontrolle: jedes SELECT — und jedes INSERT, UPDATE, DELETE — wird von Hand in Mapper-XML oder Annotationen geschrieben, plus eine resultMap pro Ergebnisform. Ehrlich, aber die Schreibseite ist reine Fron.

Modelet

Du schreibst nur die Abfragen

SELECTs sind pures ANSI-SQL in deinem Code. Die Schreibseite wird aus dem TxnMode-Zustand der Entität generiert — Schlüssel zurückgeschrieben, Audit-Felder gesetzt. ~2.000 Zeilen; an einem Nachmittag gelesen.

Der entscheidende Unterschied

Auf der Schreibseite trennen sich die Wege

Dieselbe Anforderung in allen dreien: ein Buch anlegen, dann den Preis erhöhen. Schau, wer tippt.

Modelet2 lines per write
Book book = new Book();
book.setTitle("Modelet in Action");
model.save(book);                  // INSERT — id written back, audit stamped

book.setPrice(new BigDecimal("499"));
model.save(book);                  // UPDATE — same call, TxnMode decides
MyBatisevery column, by hand
<!-- BookMapper.xml — you write and maintain every statement -->
<insert id="insertBook" useGeneratedKeys="true" keyProperty="id">
  INSERT INTO book (bookName, price) VALUES (#{title}, #{price})
</insert>
<update id="updateBook">
  UPDATE book SET bookName = #{title}, price = #{price} WHERE id = #{id}
</update>

// Java — plus a mapper interface to keep in sync
bookMapper.insertBook(book);
bookMapper.updateBook(book);
Hibernatestate machine you must learn
session.persist(book);             // transient → managed
book.setPrice(new BigDecimal("499"));
// no save call — dirty checking emits an UPDATE at flush… eventually.
// detached instance? use merge(). outside the session? LazyInitializationException.

Eine neue Spalte berührt in MyBatis die Tabelle, das XML-Insert, das XML-Update und den Mapper. In Modelet nur Tabelle und Entitätsfeld — das SQL erzeugt sich neu.

Ehrliche Anmerkungen

Wann die anderen gewinnen

Hibernate

Ein tiefes Domänenmodell mit komplexen Objektgraphen, Vendor-Portabilität und Second-Level-Cache — und ein Team, das JPA fließend spricht. Das ist Hibernates Heimspiel.

MyBatis

Du brauchst auch auf der Schreibseite handoptimiertes SQL — Bulk-Upserts, Vendor-Hints, Stored Procedures — oder die dynamischen SQL-Tags (<if>, <foreach>) passen zu deinen Abfragemustern.

Modelet

Dein Team spricht fließend SQL und will am ersten Tag liefern: Abfragen bleiben in deinem SQL, Schreibvorgänge generieren sich selbst, und das ganze Framework passt in den Kopf.

Modelet · Abfragen gehören zu SQL, Schreibvorgänge zum Framework
