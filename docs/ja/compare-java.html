Modelet vs MyBatis & Hibernate

Hibernate は SQL をすべて隠し、MyBatis は SQL をすべて書かせる。Modelet は仕事を然るべき場所で分割します。

三者が答えるのは同じ問い —— オブジェクトはどう行と出会うのか。違いはただ一つ、SQL を誰が書くか。以下、率直な比較です。

三つの哲学

あなたの SQL、誰が書く?

Hibernate

SQL をすべて隠す

マッピングと関連を記述すれば、フレームワークが flush 時に SQL を導出します。複雑なドメイングラフには強力 —— その代償は HQL/JPQL、セッションのライフサイクル、lazy プロキシ、dirty checking の不意打ちです。

MyBatis

SQL をすべて書かせる

完全な制御:すべての SELECT も —— そしてすべての INSERT、UPDATE、DELETE も —— mapper XML かアノテーションに手書きし、結果の形ごとに resultMap を用意します。誠実ですが、書き込み側は純然たる苦役です。

Modelet

書くのはクエリだけ

SELECT はコード内の素の ANSI SQL。書き込み側はエンティティの TxnMode 状態から生成され、主キーは書き戻され、監査項目も自動付与。約 2,000 行、半日で読み切れます。

決定的な違い

人生が分かれるのは、書き込み側

要件は同じ:本を 1 冊登録し、値上げする。誰がタイプしているかを見てください。

Modelet2 lines per write
Book book = new Book();
book.setTitle("Modelet in Action");
model.save(book);                  // INSERT — id written back, audit stamped

book.setPrice(new BigDecimal("499"));
model.save(book);                  // UPDATE — same call, TxnMode decides
MyBatisevery column, by hand
<!-- BookMapper.xml — you write and maintain every statement -->
<insert id="insertBook" useGeneratedKeys="true" keyProperty="id">
  INSERT INTO book (bookName, price) VALUES (#{title}, #{price})
</insert>
<update id="updateBook">
  UPDATE book SET bookName = #{title}, price = #{price} WHERE id = #{id}

// Java — plus a mapper interface to keep in sync
bookMapper.insertBook(book);
bookMapper.updateBook(book);
Hibernatestate machine you must learn

// no save call — dirty checking emits an UPDATE at flush… eventually.

MyBatis でカラムを 1 つ足すと、テーブル、XML の insert、XML の update、mapper に手が入ります。Modelet ならテーブルとエンティティのフィールドだけ —— SQL は自動的に生え変わります。

率直な注記

他が勝つ場面

Hibernate

深いドメインモデル、複雑なオブジェクトグラフ、ベンダー可搬性と二次キャッシュ —— そして JPA に習熟したチーム。そこは Hibernate のホームです。

MyBatis

書き込み側まで手調整の SQL が要る場合 —— バルク upsert、ベンダーヒント、ストアドプロシージャ —— あるいは動的 SQL タグ(<if>、<foreach>)がクエリパターンに合う場合。

Modelet

チームが SQL に習熟していて、初日から出荷したいなら:クエリは自分の SQL のまま、書き込みは自動生成、フレームワーク全体が頭に収まります。

Modelet · クエリは SQL に、書き込みはフレームワークに
